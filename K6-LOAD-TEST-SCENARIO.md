# 🎯 K6 부하테스트 시나리오 분석

## 📋 테스트 개요
**대상 애플리케이션**: Spring Boot Petclinic (https://petclinic.bluesunnywings.com)  
**테스트 유형**: 점진적 부하 증가 테스트 (Ramp-up Load Test)  
**총 소요 시간**: 17분  
**모니터링**: Prometheus Remote Write를 통한 실시간 메트릭 수집

## 🚀 부하 테스트 단계별 시나리오

### 1단계: 워밍업 (Warm-up) - 2분
```
시간: 0분 → 2분
VU: 0 → 10명
목적: 시스템 초기화 및 기본 성능 확인
```
- 애플리케이션 JVM 워밍업
- 데이터베이스 연결 풀 초기화
- 캐시 데이터 로딩

### 2단계: 정상 부하 (Normal Load) - 5분
```
시간: 2분 → 7분
VU: 10 → 30명
목적: 일반적인 운영 환경 부하 시뮬레이션
```
- 평상시 사용자 트래픽 패턴
- 시스템 안정성 확인
- 기준 성능 지표 수집

### 3단계: 피크 부하 (Peak Load) - 3분
```
시간: 7분 → 10분
VU: 30 → 50명
목적: 최대 예상 트래픽 처리 능력 테스트
```
- 마케팅 이벤트나 특별한 상황의 트래픽 급증
- 시스템 한계점 근접 테스트
- 성능 저하 지점 확인

### 4단계: 지속 테스트 (Sustained Load) - 5분
```
시간: 10분 → 15분
VU: 50명 유지
목적: 높은 부하 상태에서의 시스템 안정성 검증
```
- 메모리 누수 확인
- GC 성능 모니터링
- 데이터베이스 연결 안정성

### 5단계: 쿨다운 (Cool-down) - 2분
```
시간: 15분 → 17분
VU: 50 → 0명
목적: 시스템 복구 및 정리 과정 모니터링
```
- 리소스 해제 확인
- 시스템 정상화 시간 측정

## 🌐 테스트 시나리오 상세

### 사용자 여정 (User Journey)
각 가상 사용자(VU)는 다음 순서로 애플리케이션을 사용:

#### 1. 메인 페이지 접속 (1초 대기)
```javascript
GET https://petclinic.bluesunnywings.com/
검증:
- HTTP 상태 코드 200
- 응답 시간 < 1000ms
```

#### 2. 수의사 목록 조회 (1초 대기)
```javascript
GET https://petclinic.bluesunnywings.com/vets
검증:
- HTTP 상태 코드 200
- 페이지에 'veterinarian' 텍스트 포함
```

#### 3. 소유자 검색 페이지 (2초 대기)
```javascript
GET https://petclinic.bluesunnywings.com/owners/find
검증:
- HTTP 상태 코드 200
```

#### 4. 에러 처리 테스트 (1초 대기)
```javascript
GET https://petclinic.bluesunnywings.com/nonexistent
검증:
- HTTP 상태 코드 404 (정상적인 에러 처리)
```

**총 사이클 시간**: 약 5초 (요청 시간 + 대기 시간)

## 📊 성능 임계값 (Thresholds)

### HTTP 응답 시간
- **P95 < 1000ms**: 95%의 요청이 1초 이내 응답
- **P99 < 2000ms**: 99%의 요청이 2초 이내 응답

### 오류율
- **실패율 < 5%**: HTTP 4xx/5xx 오류가 전체 요청의 5% 미만

### 커스텀 메트릭
- **성공률 > 95%**: 비즈니스 로직 검증 성공률 95% 이상

## 🔍 모니터링 메트릭

### K6 기본 메트릭
- `k6_vus`: 현재 활성 가상 사용자 수
- `k6_http_reqs_total`: 총 HTTP 요청 수
- `k6_http_req_duration_p50/p95/p99`: 응답 시간 백분위수
- `k6_http_req_failed_rate`: HTTP 요청 실패율

### 커스텀 메트릭
- `k6_custom_errors_total`: 비즈니스 로직 검증 실패 수
- `k6_custom_success_rate_rate`: 비즈니스 로직 성공률
- `k6_custom_response_time`: 커스텀 응답 시간 메트릭

### 애플리케이션 메트릭
- CPU 사용률: `container_cpu_usage_seconds_total`
- 메모리 사용량: `container_memory_usage_bytes`
- JVM Heap 메모리: `jvm_memory_bytes_used`

## 🎯 테스트 목적별 분석

### 1. 성능 기준선 설정
- 정상 부하(30 VU) 상태에서의 기본 성능 지표
- 평균 응답 시간, 처리량(RPS) 측정

### 2. 확장성 검증
- 부하 증가에 따른 성능 변화 패턴
- 시스템 병목점 식별

### 3. 안정성 확인
- 높은 부하 상태에서 5분간 지속 운영
- 메모리 누수, 연결 풀 고갈 등 확인

### 4. 복구 능력 테스트
- 부하 제거 후 시스템 정상화 시간
- 리소스 해제 및 GC 동작 확인

## 📈 예상 성능 패턴

### 응답 시간 변화
```
워밍업(10 VU): ~100-200ms
정상 부하(30 VU): ~200-400ms  
피크 부하(50 VU): ~400-800ms
지속 테스트: 점진적 증가 가능성
쿨다운: 빠른 정상화
```

### 시스템 리소스 사용률
```
CPU: 워밍업 10% → 정상 30% → 피크 60-80%
메모리: 점진적 증가 후 GC로 정리
네트워크: 요청량에 비례하여 증가
```

## 🚨 주의 관찰 포인트

1. **GC 패턴**: 부하 증가 시 GC 빈도 및 소요 시간
2. **데이터베이스 연결**: 연결 풀 사용률 및 대기 시간
3. **메모리 누수**: 지속 테스트 중 메모리 사용량 추이
4. **응답 시간 분포**: P95, P99 값의 급격한 증가 지점
5. **에러 발생**: 특정 부하 수준에서의 오류 패턴

이 시나리오를 통해 Petclinic 애플리케이션의 성능 특성을 종합적으로 분석하고, 운영 환경에서의 적정 부하 수준을 결정할 수 있습니다.
